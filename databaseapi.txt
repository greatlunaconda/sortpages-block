
useSelect('core') を使用して`wp_actionscheduler_actions`テーブルの値を取得する方法

結論から言うと、useSelect('core') だけでは wp_actionscheduler_actions のデータを直接取得することはできません。
なぜなら、core データストアが対象とするのは、WordPress の「標準的な REST API エンドポイント（投稿、固定ページ、カテゴリーなど）」のみだからです。Action Scheduler はカスタムテーブルであるため、core の管理外にあります。
どうしても useSelect で取得したい場合は、以下の「カスタムデータソースの登録」が必要です。
実装手順
1. REST API の rest_base を「投稿タイプ」のように偽装する
まず PHP 側で、core データストアが認識できる形式でカスタムエンドポイントを登録します。
php
// functions.php 等
add_action( 'rest_api_init', function () {
    // core データストアからアクセスできるようにエンドポイントを構成
    register_rest_route( 'wp/v2', '/as-actions', array(
        'methods'  => 'GET',
        'callback' => function() {
            global $wpdb;
            return $wpdb->get_results( "SELECT * FROM {$wpdb->prefix}actionscheduler_actions LIMIT 10" );
        },
        'permission_callback' => '__return_true', // 実際は権限チェックを推奨
    ) );
} );
コードは注意してご使用ください。

2. JavaScript 側で「エンティティ」として定義する
core データストアに「こういうデータ（エンティティ）があるよ」と教えてあげる必要があります。
javascript
import { useSelect } from '@wordpress/data';
import { addEntities } from '@wordpress/core-data';
import { dispatch } from '@wordpress/data';

// 1. core データストアに新しいエンティティを追加
dispatch('core').addEntities([
    {
        name: 'as-actions', // エンティティ名
        kind: 'root',      // ルート階層
        baseURL: '/wp/v2/as-actions', // 作成したAPIのパス
    }
]);

// 2. useSelect で呼び出す
const MyComponent = () => {
    const actions = useSelect( ( select ) => {
        return select('core').getEntityRecords('root', 'as-actions');
    }, [] );

    if ( ! actions ) return 'Loading...';

    return (
        <ul>
            { actions.map( ( action ) => (
                <li key={ action.action_id }>{ action.hook }</li>
            ) ) }
        </ul>
    );
};
コードは注意してご使用ください。

なぜこの手順が必要か？
useSelect('core') は、内部的に WordPress REST API Reference にある標準パスを前提に動作します。
Action Scheduler は独自テーブルなので、addEntities を使って 「core データの仲間」として手動登録 しない限り、getEntityRecords などの便利な関数が反応してくれません。
この方法は少し高度ですが、複数のコンポーネントで Action Scheduler のデータを同期・共有したい場合に非常に有効です。
今回取得したいデータは、単なる表示用ですか？それとも、JavaScript 側からアクションを再実行・削除するなどの操作も含みますか？

rest api によって`wp_actionscheduler_actions`テーブルの値を取得する方法
WordPress の Action Scheduler (wp_actionscheduler_actions) には、デフォルトで専用の REST API エンドポイントは用意されていません。 
そのため、このテーブルの値を取得するには、以下のいずれかの方法でカスタムエンドポイントを自作する必要があります。
1. カスタム REST API エンドポイントを作成する
register_rest_route 関数を使用して、独自の API パスを作成します。PHP 内で Action Scheduler の関数（as_get_scheduled_actions など）を呼び出すことで、テーブルデータを JSON 形式で返せます。 
実装例（functions.php などに記述）:
php
add_action( 'rest_api_init', function () {
    register_rest_route( 'my-plugin/v1', '/actions', array(
        'methods'  => 'GET',
        'callback' => 'get_as_actions_data',
        'permission_callback' => function () {
            return current_user_can( 'manage_options' ); // 管理者権限が必要
        }
    ) );
} );

function get_as_actions_data() {
    // Action Scheduler の API を使用してデータを取得
    // または $wpdb を使って直接 wp_actionscheduler_actions テーブルをクエリ
    $actions = ActionScheduler_wpPostStore::query_actions( array( 'per_page' => 10 ) );
    return rest_ensure_response( $actions );
}
コードは注意してご使用ください。

2. $wpdb を使った直接クエリ
特定のステータス（完了、待機中など）を柔軟に取得したい場合は、$wpdb を用いて直接 SQL を実行するのが効率的です。 
テーブル名: {$wpdb->prefix}actionscheduler_actions
関連テーブル: {$wpdb->prefix}actionscheduler_logs（履歴が必要な場合）
注意点
セキュリティ: Action Scheduler のデータには機密情報が含まれる可能性があるため、必ず permission_callback で適切な権限チェックを行ってください。
データの肥大化: 稼働時間の長いサイトではこのテーブルが数 GB 規模になることがあるため、取得時の LIMIT 設定は必須です。

useSelect('core') を使用して`wp_terms`テーブルの値を取得する方法
wp_terms テーブル（カテゴリーやタグ）は WordPress の標準エンティティであるため、useSelect('core') を使って追加設定なしで直接取得できます。
基本的な書き方
カテゴリー（category）やタグ（post_tag）などのタクソノミーを指定して取得します。
javascript
import { useSelect } from '@wordpress/data';

const MyTermList = () => {
    const { terms, hasResolved } = useSelect( ( select ) => {
        const query = { per_page: -1, hide_empty: false }; // 全件取得、空のタームも含む
        return {
            terms: select( 'core' ).getEntityRecords( 'taxonomy', 'category', query ),
            hasResolved: select( 'core' ).hasFinishedResolution( 'getEntityRecords', [ 'taxonomy', 'category', query ] ),
        };
    }, [] );

    if ( ! hasResolved ) return <p>読み込み中...</p>;
    if ( ! terms ) return <p>データがありません</p>;

    return (
        <ul>
            { terms.map( ( term ) => (
                <li key={ term.id }>{ term.name } (ID: { term.id })</li>
            ) ) }
        </ul>
    );
};
コードは注意してご使用ください。

取得条件の指定
getEntityRecords の第 3 引数にクエリを渡すことで、取得結果を絞り込めます。
タグを取得する場合: 'category' を 'post_tag' に変更します。
特定の投稿に紐づくタームを取得: select('core').getEntityRecords('taxonomy', 'category', { post: 123 })
親子関係のフィルタリング: parent: 0 を指定すると親カテゴリーのみ取得できます。
注意点
カスタムタクソノミーの場合: PHP 側で register_taxonomy する際に 'show_in_rest' => true が設定されている必要があります。設定されていないと useSelect から参照できません。
データの同期: useSelect は内部で Redux ストア を監視しているため、別のコンポーネントでタームが更新されると自動的に再レンダリングされます。
特定のカスタムタクソノミーを取得しようとして、データがうまく返ってこないなどの問題は発生していますか？

rest api によって`wp_terms`テーブルの値を取得する方法
wp_terms テーブルのデータ（カテゴリーやタグなど）を取得するには、WordPress 標準の Taxonomy エンドポイントを使用するのが最も簡単で確実です。 
1. 標準エンドポイントを使用する
WordPress には、wp_terms に紐づく主要なデータの取得用 URL がデフォルトで用意されています。
カテゴリー一覧: https://example.com/wp-json/wp/v2/categories
タグ一覧: https://example.com/wp-json/wp/v2/tags 
2. カスタムタクソノミーの場合
自分で作成したカスタムタクソノミーのデータを取得したい場合は、そのタクソノミーを登録する際の register_taxonomy 関数で、必ず 'show_in_rest' => true を設定する必要があります。 
エンドポイント例: https://example.com/wp-json/wp/v2/{taxonomy_slug}
※ rest_base を指定している場合は、その値が URL になります。 
3. パラメーターによる絞り込み
URL の末尾にクエリパラメーターを追加することで、取得結果を調整できます。 
特定の親を持つタームのみ: ?parent={id}
件数指定: ?per_page=100 (デフォルトは 10)
特定の名前で検索: ?search=キーワード 
4. 特定の ID の値を取得する
特定のターム 1 件の詳細を取得する場合は、URL の末尾に ID を付与します。
https://example.com/wp-json/wp/v2/categories/{id} 
もし、標準の API では取得できない特殊なカラム（独自に追加したカラムなど）を含めたい場合は、前述の Action Scheduler の例と同様に カスタムエンドポイントを自作 して $wpdb で直接クエリを発行する必要があります。

useSelect('core') を使用して`wp_termmeta`テーブルの値を取得する方法
wp_termmeta テーブルの値を useSelect('core') で取得するには、そのメタデータが REST API に公開されていることが必須条件です。デフォルトではセキュリティ上の理由から、メタデータは API に含まれません。
以下の2ステップで実装します。
1. PHP側：メタデータを REST API に登録する
まず、register_term_meta を使用して、特定のメタキーを REST API で読み書きできるように許可します。
php
add_action( 'init', function() {
    register_term_meta( 'category', 'my_custom_meta_key', array(
        'show_in_rest' => true, // これを true にするのがポイント
        'single'       => true,
        'type'         => 'string',
    ) );
} );
コードは注意してご使用ください。

2. JS側：useSelect で取得する
メタデータは、取得したタームオブジェクト内の meta プロパティに格納されます。
javascript
import { useSelect } from '@wordpress/data';

const CategoryMeta = ( { categoryId } ) => {
    const category = useSelect( ( select ) => {
        // getEntityRecord (単数形) を使用して特定のカテゴリーを取得
        return select( 'core' ).getEntityRecord( 'taxonomy', 'category', categoryId );
    }, [ categoryId ] );

    // メタデータの取得
    const metaValue = category?.meta?.['my_custom_meta_key'];

    if ( ! category ) return 'Loading...';

    return (
        <div>
            <p>カテゴリー名: { category.name }</p>
            <p>カスタムメタ値: { metaValue || '未設定' }</p>
        </div>
    );
};
コードは注意してご使用ください。

重要なポイント
個別登録が必要: wp_termmeta テーブルにある全ての値が自動で取れるわけではありません。取得したいキー（meta_key）ごとに register_term_meta で show_in_rest => true を設定する必要があります。
一括取得の場合: getEntityRecords (複数形) で取得した場合も、各タームオブジェクトの中に meta フィールドが含まれます。
ACF を使っている場合: Advanced Custom Fields を使用しているなら、設定画面から「Show in REST API」を ON にするだけで、acf プロパティ経由で取得可能になります。
特定のプラグインが作成したメタデータや、ACFなどで作成したフィールドを取得しようとしていますか？

useSelect('core') を使用して'wp_commentmeta'テーブルの値を取得する方法
wp_commentmeta テーブルの値を useSelect('core') で取得する場合、投稿（post）と同様に PHP 側で REST API への露出設定を行い、その後 JS 側で comment エンティティを介してアクセスします。
1. PHP 側：コメントメタの登録
コメントのメタデータはデフォルトでは REST API に含まれません。register_meta を使用して、show_in_rest を true に設定する必要があります。 
php
// functions.php などに記述
add_action( 'init', function() {
    register_meta( 'comment', 'your_meta_key', array(
        'type'         => 'string',
        'single'       => true,
        'show_in_rest' => true, // これが必須
    ) );
} );
コードは注意してご使用ください。

2. JS 側：useSelect による取得
メタデータが登録されると、select('core').getEntityRecords で取得したコメントオブジェクトの meta プロパティ内に値が含まれるようになります。 
javascript
import { useSelect } from '@wordpress/data';

const CommentMetaViewer = ( { commentId } ) => {
    const comment = useSelect( ( select ) => {
        // 'root' kind の 'comment' エンティティから特定のIDを取得
        return select( 'core' ).getEntityRecord( 'root', 'comment', commentId );
    }, [ commentId ] );

    // メタデータの取得
    const metaValue = comment?.meta?.['your_meta_key'];

    return (
        <div>
            コメントメタの値: { metaValue || 'データなし' }
        </div>
    );
};
コードは注意してご使用ください。

注意点
テーブル直接アクセスではない: useSelect は SQL を発行するのではなく、REST API エンドポイント (/wp/v2/comments) を叩きます。そのため、API が返さないデータにはアクセスできません。
getEntityRecords の利用: 複数のコメントのメタを一覧したい場合は getEntityRecords( 'root', 'comment' ) を使用します。

useSelect('core') を使用して`wp_comments`テーブルの値を取得する方法
wp_comments テーブルの情報は、WordPress 標準の comment エンティティとして core データストアに登録されています。そのため、特別な設定なしで useSelect から直接取得可能です。
基本的な取得方法
getEntityRecords を使用します。第1引数に 'root'、第2引数に 'comment' を指定するのがポイントです。
javascript
import { useSelect } from '@wordpress/data';

const MyComments = () => {
    const { comments, hasResolved } = useSelect( ( select ) => {
        // クエリ条件（例：最新10件、承認済みのみ）
        const query = { per_page: 10, status: 'approve' };
        
        return {
            comments: select( 'core' ).getEntityRecords( 'root', 'comment', query ),
            hasResolved: select( 'core' ).hasFinishedResolution( 'getEntityRecords', [ 'root', 'comment', query ] ),
        };
    }, [] );

    if ( ! hasResolved ) return <p>読み込み中...</p>;

    return (
        <ul>
            { comments?.map( comment => (
                <li key={ comment.id }>
                    { comment.author_name }: { comment.content.raw }
                </li>
            ) ) }
        </ul>
    );
};
コードは注意してご使用ください。

フィルタリングのバリエーション
クエリパラメーターを書き換えることで、特定の条件に絞った wp_comments の値を取得できます。
特定の投稿に紐づくコメント: { post: 投稿ID }
特定のユーザーのコメント: { author: ユーザーID }
親子関係（返信）の取得: { parent: 親コメントID }
未承認コメントの確認: { status: 'hold' }（※閲覧権限が必要です）
補足：commentmeta を取得する場合
wp_commentmeta の値も一緒に取得したい場合は、事前に PHP 側で register_meta を使い、show_in_rest を true に設定しておく必要があります。

useSelect('core') を使用して`wp_gla_attribute_mapping_rules`テーブルの値を取得する方法
wp_gla_attribute_mapping_rules は、Google Listing & Ads プラグイン固有のカスタムテーブルです。
WordPress の core データストアは、標準テーブル（投稿、コメント、タームなど）しか認識しません。そのため、useSelect('core') でこのテーブルを取得するには、「カスタム REST API の作成」 と 「core データストアへのエンティティ登録」 の両方が必要になります。
手順 1: PHP でカスタム API を作成
このテーブルの内容を JSON で返すエンドポイントを自作します。
php
// functions.php またはプラグインファイル
add_action( 'rest_api_init', function () {
    register_rest_route( 'gla/v1', '/mapping-rules', array(
        'methods'  => 'GET',
        'callback' => function() {
            global $wpdb;
            $table_name = $wpdb->prefix . 'gla_attribute_mapping_rules';
            // テーブルから全件取得
            return $wpdb->get_results( "SELECT * FROM {$table_name}" );
        },
        'permission_callback' => function () {
            return current_user_can( 'manage_options' );
        }
    ) );
} );
コードは注意してご使用ください。

手順 2: JS で core データストアに登録
useSelect('core') で呼び出せるように、クライアント側でエンティティとして追加します。
javascript
import { useSelect, dispatch } from '@wordpress/data';
import { useEffect } from '@wordpress/element';

// エンティティの登録（一度だけ実行）
dispatch('core').addEntities([
    {
        name: 'mapping-rules',
        kind: 'gla-plugin', // 任意のグループ名
        baseURL: '/gla/v1/mapping-rules',
    }
]);

const MappingRulesList = () => {
    const rules = useSelect( ( select ) => {
        // 第1引数に kind、第2引数に name を指定
        return select( 'core' ).getEntityRecords( 'gla-plugin', 'mapping-rules' );
    }, [] );

    if ( ! rules ) return <p>Loading Rules...</p>;

    return (
        <ul>
            { rules.map( rule => (
                <li key={ rule.id }>{ rule.attribute } → { rule.source }</li>
            ) ) }
        </ul>
    );
};
コードは注意してご使用ください。

なぜこの手順が必要か？
非標準テーブル: WordPress の Core Data セレクター は、wp_posts や wp_comments のように API パスが標準化されているものしか自動で扱いません。
エンティティ定義: addEntities を使うことで、getEntityRecords などの標準的な コアデータ関数 を使ってカスタムデータを管理（キャッシュ、取得、更新）できるようになります。

wp_gla_budget_recommendations も Google Listing & Ads プラグインの専用カスタムテーブルであるため、先ほどと同様に 「REST API の自作」 と 「core データストアへのエンティティ登録」 が必要です。
WordPress の標準機能には含まれていないため、以下の手順で「core データの仲間」として定義します。
手順 1: PHP で API エンドポイントを作成
wp_gla_budget_recommendations テーブルのデータを返すエンドポイントを登録します。

add_action( 'rest_api_init', function () {

        'permission_callback' => function () {

手順 2: JS でエンティティを登録して useSelect で取得
dispatch('core').addEntities を使って、core データストアからアクセス可能にします。
javascript
import { useSelect, dispatch } from '@wordpress/data';

// 1. coreデータストアにカスタムエンティティを紐付け
dispatch('core').addEntities([
    {
        name: 'budget-recommendations',
        kind: 'gla-plugin',
        baseURL: '/gla/v1/budget-recommendations',
    }
]);

// 2. コンポーネント内で使用
const BudgetRecommendations = () => {
    const { recommendations, hasResolved } = useSelect( ( select ) => {

            recommendations: select( 'core' ).getEntityRecords( ...selectorArgs ),

        };

    if ( ! hasResolved ) return <p>Loading recommendations...</p>;

    return (
        <ul>
            { recommendations?.map( rec => (
                <li key={ rec.id }>推奨予算: { rec.daily_budget }</li>
            ) ) }
        </ul>
    );
};
コードは注意してご使用ください。

キャッシュの利点: 一度 addEntities で登録してしまえば、useSelect('core') は自動的に ブラウザ内キャッシュ を管理してくれるため、パフォーマンスが向上します。

add_action( 'rest_api_init', function () {

wp_options テーブルのデータは、WordPress の Settings API を介して core データストアに統合されています。

useSelect('core') を使用して`wp_posts`テーブルの値を取得する方法
wp_posts テーブルのデータは、WordPress REST API の中核（Core）であるため、useSelect('core') を使って最も簡単に取得できるエンティティです。
基本的な取得方法
getEntityRecords を使用します。第1引数に 'postType'、第2引数に投稿タイプ（post, page, またはカスタム投稿タイプ名）を指定します。
javascript
import { useSelect } from '@wordpress/data';

const PostList = () => {
    const { posts, hasResolved } = useSelect( ( select ) => {
        // 取得条件（例：最新5件、公開済み）
        const query = { per_page: 5, status: 'publish' };
        
        return {
            posts: select( 'core' ).getEntityRecords( 'postType', 'post', query ),
            hasResolved: select( 'core' ).hasFinishedResolution( 'getEntityRecords', [ 'postType', 'post', query ] ),
        };
    }, [] );

    if ( ! hasResolved ) return <p>読み込み中...</p>;

    return (
        <ul>
            { posts?.map( post => (
                <li key={ post.id }>{ post.title.rendered }</li>
            ) ) }
        </ul>
    );
};
コードは注意してご使用ください。

よく使うセレクターの使い分け
用途に応じて、core データストア内の異なるセレクターを使い分けます。
セレクター	用途
getEntityRecords	条件に合う複数の投稿を一覧で取得する時に使用
getEntityRecord	ID を指定して特定の1件を取得する時に使用
getEditedPostAttribute	ブロックエディターで編集中の投稿（未保存の値を含む）を取得する時に使用 ※core/editor ストア推奨
条件による絞り込み（クエリパラメーター）
第3引数のオブジェクトで、SQLの WHERE や ORDER BY に相当する指定が可能です。
特定のカテゴリーで絞り込む: { categories: [ ID ] }
特定の著者で絞り込む: { author: ユーザーID }
検索ワードを指定する: { search: 'キーワード' }
固定ページを取得する: 第2引数を 'page' に変更
注意点：カスタム投稿タイプの場合
自作したカスタム投稿タイプを取得したい場合は、PHP 側の register_post_type の引数で 'show_in_rest' => true を設定していることを確認してください。これがないと useSelect でデータを引っ張ることができません。

useSelect('core') を使用して `wp_wc_admin_notes`テーブルの値を取得する方法
wp_wc_admin_notes は WooCommerce Admin（マーケティング通知や通知メッセージ）のカスタムテーブルであるため、WordPress の core データストアにはデフォルトで登録されていません。
しかし、WooCommerce はこのテーブル用の REST API エンドポイントを標準で提供しているため、core データストアに「エンティティ」として定義を追加することで、useSelect('core') を使った取得が可能になります。
実装手順
1. core データストアにエンティティを登録する
JavaScript 側で、WooCommerce のノート用エンドポイントを core の仲間として認識させます。
javascript
import { useSelect, dispatch } from '@wordpress/data';

// 1. core データストアにエンティティ定義を追加 (一度だけ実行)
dispatch('core').addEntities([
    {
        name: 'admin-notes',     // 任意の名前
        kind: 'wc/v3',          // 任意のグループ
        baseURL: '/wc/v3/admin/notes', // WooCommerce 標準のAPIパス
    }
]);
コードは注意してご使用ください。

2. useSelect で取得する
登録した名前（admin-notes）とグループ（wc/v3）を指定して呼び出します。
javascript
const AdminNotesList = () => {
    const { notes, hasResolved } = useSelect( ( select ) => {
        const args = [ 'wc/v3', 'admin-notes' ];
        return {
            notes: select( 'core' ).getEntityRecords( ...args ),
            hasResolved: select( 'core' ).hasFinishedResolution( 'getEntityRecords', args ),
        };
    }, [] );

    if ( ! hasResolved ) return <p>読み込み中...</p>;

    return (
        <ul>
            { notes?.map( note => (
                <li key={ note.id }>
                    <strong>{ note.title }</strong>: { note.content }
                </li>
            ) ) }
        </ul>
    );
};
コードは注意してご使用ください。

注意点
API の存在確認: このエンドポイントは WooCommerce REST API Documentation に基づいています。WooCommerce が有効である必要があります。
認証と権限: 通常、管理通知の取得には manage_woocommerce 権限が必要です。REST API 経由で取得する場合、適切にログインしている必要があります。
テーブル内容: wp_wc_admin_notes テーブルには、通知のタイトル、内容、ステータス（既読/未読）などが保存されています。
